SAML.

Security Assertion Markup Language — how single sign-on actually works, how attackers tear it apart, and why Golden SAML is the attack you have to know cold.

Normal Flow Okta → Slack Attacks Golden SAML SolarWinds CVEs Defense

SAML (Security Assertion Markup Language) exists so you log in once at a central authority and every connected app trusts that login — no separate password per app, no duplicate user databases.

01Normal scenario

Three parties, one signed XML blob. The IdP vouches for the user; the SP trusts the signature.

Principal
The user being authenticated.
SP — Service Provider
The application the user wants to access (Slack, Salesforce, AWS console).
IdP — Identity Provider
The authentication authority (Okta, Azure AD, ADFS = Active Directory Federation Services, Ping, Auth0).
AuthnRequest
XML message from the SP asking the IdP to authenticate the user.
Assertion
Signed XML blob from the IdP that says "this is user X, authenticated at time Y, allowed to use SP Z".
ACS — Assertion Consumer Service
The SP endpoint that receives the posted assertion.

Everything hangs on XML-DSig (XML Digital Signature, RFC 3275). The SP verifies the assertion came from the IdP using the IdP's public certificate — registered when the SSO integration was configured — without having to trust the browser that carried it.

03Real-life — Okta → Slack

You open slack.com at an enterprise tenant. Slack has no password for you — it generated a SAML AuthnRequest and 302-redirected your browser to company.okta.com/app/slack/sso/saml.

You authenticate with Okta: password plus push MFA (Multi-Factor Authentication). Okta builds a SAML Response containing an Assertion with NameID=ag@company.com, group memberships, and role attributes, signs it with Okta's private certificate, and sends your browser a form that auto-POSTs back to slack.com/saml/acs.

Slack then validates everything:

  • Signature against Okta's public cert (pinned when SSO was configured)
  • Audience matches Slack's entity ID
  • Recipient / Destination matches the ACS URL
  • NotOnOrAfter is within a 5-minute window
  • InResponseTo matches the AuthnRequest Slack actually issued

All green → Slack drops you into the workspace with the right org and roles. You never typed a Slack password.

04Attacker perspective

SAML's attack surface is unusually rich because assertions are XML, signatures are applied to XML subtrees, and the same document is parsed twice — once for signature verification, once for claim extraction. That seam is where the bugs live.

  1. 1. XML Signature Wrapping (XSW)
    XML allows duplicate element IDs. You wrap the legitimately signed assertion inside a new envelope and inject a forged assertion at a different location. The signature-verification parser reads the original element and returns "valid"; the claim-extraction parser reads the injected one and returns attacker-controlled claims. Same-Document Reference confusion. SAML Raider Burp extension has 8 pre-built wrapping variants.
  2. 2. Signature exclusion
    Strip the <ds:Signature> element entirely. Misconfigured SPs accept unsigned assertions when no signature is present (instead of rejecting). Surprisingly common in homegrown integrations.
  3. 3. Comment injection in NameID
    <NameID>admin@victim.com<!---->.attacker.com</NameID>. Some XML libraries preserve comments during signature canonicalization but the claim-extraction code truncates at the comment. Signature verifies against the full string; the SP reads admin@victim.com.
  4. 4. Algorithm downgrade
    Force SignatureMethod to MD5, SHA1, or none if the library tolerates it.
  5. 5. Missing claim validation
    SP doesn't enforce Audience / Recipient / NotOnOrAfter / InResponseTo. Replay assertions across SPs, or long after issuance.
  6. 6. XXE in the SAML parser
    If DTD (Document Type Definition) processing and external entity resolution are enabled, the assertion becomes an XXE (XML External Entity) vector — file reads, SSRF, the usual XXE consequences.

05Golden SAML — the nuclear option

Once the IdP's private signing key is in the attacker's hands, SAML's entire trust model collapses. Assertions can be forged for any user, any role, any SP — and the signatures are cryptographically indistinguishable from genuine ones.

Real incident — SolarWinds (December 2020)

APT29 (Nobelium / Cozy Bear, Russian SVR) used the SolarWinds Orion supply-chain compromise to land inside victim networks, then pivoted onto ADFS servers and exfiltrated the token-signing certificate. With the private key in hand they forged SAML tokens for arbitrary users — including Global Admins — and authenticated to Microsoft 365 (Exchange, Teams, SharePoint) and Azure AD without ever touching the real IdP.

No user typed a password, no MFA prompt fired, no IdP login event was generated. The signatures were cryptographically valid and Microsoft had no protocol-level way to distinguish forged from genuine. CyberArk researchers named and demonstrated the technique in 2017 — three years before it was used at this scale.

Notable CVEs (quote these in reports)

CVE-2017-11427 — python-saml, OneLogin, and others. Comment-truncation NameID bypass; auth as any user with a one-character payload.

CVE-2024-45409 — ruby-saml, September 2024. Improper signature validation; full auth bypass. Hit GitLab and every downstream Ruby SAML consumer.

06Defense / fix for client reports

SP side

  • Require signatures on both <Response> AND <Assertion> — defense in depth against XSW
  • Validate against a single pinned IdP certificate, not any cert in a chain
  • Reject unsigned assertions outright; reject SignatureMethod=none
  • Use a SAML library where signature verification and claim extraction reference the same DOM node — parser mismatch is the root cause of XSW
  • Strictly validate Audience, Recipient / Destination, NotBefore / NotOnOrAfter (≤5 min), and InResponseTo
  • Track assertion IDs for replay protection across the validity window
  • Disable DTD processing and external entity resolution — kills XXE
  • Parse NameID with a library that strips or ignores comments

IdP side (anti-Golden SAML)

  • Store token-signing keys in an HSM (Hardware Security Module) — private key non-exportable
  • On ADFS: restrict signing-cert export to tier-0 admins; alert on certutil -exportpfx or DKM extraction
  • Rotate signing certificates on a defined schedule
  • SIEM (Security Information and Event Management) correlation: any SP SAML login without a matching IdP auth event = forged assertion
  • Monitor for anomalous assertions — users who never log in, foreign geolocation, issuance bursts
  • Treat ADFS / federation servers as tier-0: full EDR (Endpoint Detection and Response), segmentation, no shared credentials with general Windows admins
HTB Barrier note. The box revolves around XSW — specifically variants 1–4. Load the SAML Raider Burp extension, intercept the SAML Response at the ACS POST, and iterate through the wrapping payloads until the SP accepts an assertion for a different user. Comment injection in NameID is the secondary path if the library is python-saml-old.
