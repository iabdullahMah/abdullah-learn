01

IBM WebSphere — the big picture

WebSphere is not one product — it's IBM's family of enterprise Java products that work together to run large-scale web applications. When someone says "we run WebSphere," they mean they're running some combination of these components: WAS (the engine), IHS (the web server), Portal (the UI framework), DataPower (the gateway), and DMGR (the admin brain).

IBM built this for banks, governments, airlines — anyone who needs "if this goes down, people lose millions." Everything is designed around high availability and separation of concerns.

WebSphere product family — not one product, a collection
Restaurant analogy

Think of the whole WebSphere stack as a large professional restaurant. There's a waiter (IHS) who greets customers and takes orders. A display kitchen (Portal nodes) where you see your food being plated. A back kitchen (WAS) where the real cooking happens. A bouncer at the VIP entrance (DataPower) who checks credentials. And a restaurant manager (DMGR) who controls everything from the office. Each piece has one job. Together they serve thousands of customers.

02

WAS — WebSphere Application Server

WAS is the core engine that actually runs Java code. When someone submits a form, makes an API call, or triggers a bank transfer — WAS is where that logic executes. It's a Java EE (now Jakarta EE) application server, meaning it provides a standardized environment for running enterprise Java applications.

What it does

Runs Servlets, JSPs, EJBs. Manages database connections (JDBC pools), transactions (commit/rollback), security (JAAS), and messaging (JMS queues).

Why it's used

Raw Java can handle HTTP, but WAS gives clustering, failover, transaction management, connection pooling, and an admin console — you don't build these yourself for millions of banking transactions.

PT surface

Admin console /ibm/console on ports 9043/9060. Java deserialization (CVE-2015-7450). JNDI injection. EJB interfaces. IIOP/RMI ports. Default credentials.

How a request flows inside a WAS server instance
Restaurant analogy

WAS is the commercial kitchen. Not a camping stove — a full industrial kitchen with prep stations (servlets), specialized chefs (EJBs for business logic), a walk-in fridge with organized shelves (JDBC connection pool), and a fire suppression system (transaction manager that rolls back if something goes wrong). You could cook on a camping stove, but when you're serving 5,000 banking transactions a minute, you need the full commercial setup.

03

JVM — Java Virtual Machine

Every WAS instance, every portal node, every cluster member runs inside a JVM. A JVM is a single java process running on the operating system. When WebSphere starts a "server instance," it's literally starting a java process with specific memory settings (heap size, thread count, etc.).

One JVM = one OS process = one "server" in WebSphere terminology
Key internals

Heap — where sessions and objects live in memory. Thread pool — each HTTP request gets one thread; exhaust them and the server stops responding. Class loader — loads your .war/.ear application code. Connection pool — pre-opened DB connections shared across requests.

PT relevance

Crash a JVM via deserialization = only that instance dies, others keep running. Exhaust the thread pool via slow HTTP = denial of service on that JVM. Trigger excessive object creation = OutOfMemoryError crash. Sessions in heap = session fixation/hijacking surface.

Restaurant analogy

The JVM is one chef with their own workstation. The chef has a cutting board (heap memory), can handle a specific number of orders at once (thread pool), follows recipes from a cookbook (application code loaded by class loader), and grabs pre-prepped ingredients from containers (connection pool to DB). If you overwhelm one chef with 50 orders, they freeze — but the other chefs at other stations keep working independently.

04

IHS — IBM HTTP Server

IHS is IBM's rebranded Apache HTTP Server with added IBM modules. It sits at the very front — the first thing that processes an HTTP request after the load balancer. In typical enterprise deployments you'll see multiple IHS instances behind an F5 or similar LB.

What it does

Serves static content (CSS, JS, images) directly. Handles SSL/TLS termination. Runs the WebSphere Plugin which reads plugin-cfg.xml to route dynamic requests to the right cluster node.

The plugin magic

plugin-cfg.xml contains a complete map of every cluster, node, IP, and port. The IHS plugin reads this and decides: "this request for /portal/app goes to Cluster C1, Node B" (least connections algorithm).

PT gold

If plugin-cfg.xml is accessible (try /plugin-cfg.xml or path traversal), you get the entire internal network map — every node IP, port, and cluster topology. Also check: IHS version disclosure, mod_rewrite bypasses, default error pages.

IHS routes static content directly, dynamic content to clusters via plugin
Restaurant analogy

IHS is the waiter. Customer walks in and says what they want. If it's something on the counter already (bread, water = static file), the waiter grabs it directly without going to the kitchen. If it's a cooked dish (dynamic request), the waiter checks the seating chart (plugin-cfg.xml) to know which kitchen line handles which section, and sends the order to the right one.

05

Clusters — why they exist

A cluster is a group of identical JVM instances running the exact same application, working together as one logical unit. Typical enterprise setup: 2 clusters per IHS (C1 and C2), each cluster with 3 nodes (A, B, C).

High availability

If Node A dies, Nodes B and C keep serving. The WebSphere plugin in IHS detects the dead node and stops sending it traffic. Users never notice.

Scalability

1 JVM handles ~200 concurrent users. Need 600? Run 3 nodes in a cluster. Need 1,200? Add more. Scale horizontally without rewriting code.

Session failover

If user_42's session is on Node A and it crashes, the session was already replicated to Node B via DRS (Data Replication Service). User keeps working.

3 identical nodes in a cluster with session replication for failover
Restaurant analogy

A cluster is three identical kitchen lines side by side. Each line has the same equipment, same recipes, same menu. If kitchen line 1 catches fire, lines 2 and 3 keep cooking. Friday night rush? All 3 lines work simultaneously. A customer's order ticket (session) is copied to a shared board — any line can pick it up if the original goes down.

06

Nodes — and what "code viewable" means

In typical enterprise architectures, the cluster nodes run WebSphere Portal — the presentation/UI layer. The WAS servers run the backend logic. When ops folks say the nodes have "code viewable," they mean the portal nodes contain the presentation layer — what the user actually sees.

Portal node = presentation (viewable code) vs WAS = backend (business logic)

Here's what lives on the portal node — the "code viewable" parts:

  • Themes — the HTML/CSS/JS that controls how pages look
  • Portlets — small Java web components (widgets) that each handle a piece of the UI (account summary portlet, transaction history portlet, transfer form portlet)
  • JSP pages — server-side templates that generate the HTML sent to the browser
  • Actions — when a user clicks a button, the portlet fires an "action" that makes a SOAP/REST/EJB call to the WAS backend where the real logic lives

The node is the presentation layer. It knows how to display things and how to ask WAS to do things. It does NOT contain the business logic itself.

Restaurant analogy

The portal node is the open display kitchen — the fancy one customers can see from their table. You watch the chef plate your dish beautifully (themes, UI). But the actual cooking (raw meat prep, sauce reduction, oven work) happened in the back kitchen you can't see (WAS). The display chef's job is to assemble, present, and hand you the final plate. When you say "add extra sauce," the display chef yells to the back kitchen (service call), they do it, and send it forward.

07

DataPower — the gateway

IBM DataPower is a purpose-built gateway appliance (hardware or virtual). It's not a general-purpose server — it's specifically designed to sit at the network edge and handle security, transformation, and routing of messages at wire speed. The OS is locked down, hardened, no shell access by default.

Every request that passes through DataPower goes through a processing policy — a pipeline of steps. If any step fails, the request is rejected before it ever reaches WAS.

Every request passes through this pipeline — if any step fails, the request never reaches WAS
Restaurant analogy

DataPower is the health inspector permanently stationed at the delivery entrance. Every food delivery (API request) comes through this inspector. They check: is the delivery company authorized (authentication)? Is the food properly labeled (schema validation)? Is it at the right temperature (threat protection)? Is it within the daily order limit (rate limiting)? Only after passing ALL checks does the food enter the kitchen. Anything that fails gets sent back at the door.

08

ESP — Enterprise Service Proxy

ESP is not a separate product — it's a role that DataPower plays. When DataPower is configured as an ESP, it acts as an internal service proxy that mediates communication between services. A typical ESP has two jobs:

Job 1: inbound mobile routing

Mobile request comes from DataPower DMZ → ESP receives it → ESP routes it to the correct WAS server. The ESP is the internal traffic cop deciding which WAS instance handles the request.

Job 2: outbound proxy for nodes

When a portal cluster node needs to call an external API (credit bureau, payment processor), it doesn't connect to the internet directly. It sends through the ESP, and the ESP makes the outbound call. Nodes never have direct internet access.

ESP dual role: inbound API routing + outbound proxy for cluster nodes
Restaurant analogy

The ESP is the expediter (called "the pass" in a professional kitchen). Two jobs: (1) Delivery orders from Uber Eats (mobile API) come to the expediter, who reads the order and assigns it to the right station in the back kitchen (WAS). (2) When a chef (cluster node) needs a special ingredient from an external supplier, the chef doesn't leave the kitchen — they tell the expediter, and the expediter calls the supplier and brings it back. Chefs never leave the kitchen (nodes never touch the internet).

09

Complete request flow — both paths

Now that you understand every piece, here's how a single request flows through the entire stack for both entry points. Two doors, one kitchen, one database.

Web portal user clicks "View my balance"

1

Browser sends HTTPS request to portal.bank.com

2

F5 Portal picks one of 3 IHS servers (load balancing)

3

IHS serves static CSS/JS directly, then the WebSphere Plugin reads plugin-cfg.xml and routes the dynamic request to Cluster C1, Node B

4

Node B (running WebSphere Portal) loads the account portlet, renders the theme/JSP, and the portlet's action calls the backend via SOAP/REST

5

That service call goes to a WAS server which queries the Database, gets the balance, returns it

6

Node B assembles the final HTML page and sends it back through IHS → F5 → browser

Mobile user checks balance in the app

1

App sends HTTPS API request to api.bank.com

2

F5 API routes to DataPower DMZ

3

DP DMZ terminates SSL, validates auth token, checks JSON schema, runs threat protection

4

DP DMZ forwards to DataPower ESP (internal)

5

ESP routes to the correct WAS server

6

WAS queries the Database, processes JSON response, sends it back

7

Response flows back: WAS → ESP → DP DMZ → F5 API → mobile app

Critical insight

The mobile path never touches IHS or the portal nodes. Completely different entry path, same WAS backend, same database. Two doors, one kitchen. This means you test both paths independently — a vulnerability in the portal path doesn't mean it exists in the API path, and vice versa. The F5s likely have different WAF rules, the authentication mechanisms may differ, and the DataPower pipeline adds a layer the web path doesn't have.

Architecture key points

Two separate entry points

F5 Portal (web) and F5 API (mobile) are completely different F5 instances — different VIPs, likely different WAF rules, different SSL configs. Test each independently. A bypass on one doesn't mean it exists on the other.

Paths diverge then converge

Web traffic goes through IHS → Portal clusters. Mobile goes through DataPower DMZ → ESP. Both paths converge at WAS servers and the same database. The same SQLi in WAS is reachable from both doors.

DataPower is mobile-only

The web portal path never touches DataPower (except outbound via ESP). The DataPower pipeline (SSL, auth, schema, threat protection) only sits on the mobile/API path. Web portal traffic bypasses all of those checks.

ESP is the only exit

Cluster nodes cannot reach the internet directly. All outbound calls go through DataPower ESP. This is a security control but also an SSRF opportunity — if you control what service the ESP calls, you may pivot internally.

Pre-engagement questions for the client

  • What WebSphere version and fix pack are the clusters and WAS running?
  • Is the WAS admin console (/ibm/console) accessible from the testing network?
  • Are DataPower management interfaces (port 5550 XML mgmt, 9090 web GUI) in scope?
  • Is plugin-cfg.xml accessible via the web? (common misconfig)
  • Are the two F5s (Portal vs API) configured with different iRules / WAF policies?
  • What authentication does the mobile API path use — OAuth, API keys, mutual TLS?
  • Is DMGR (:9060) reachable from the testing VLAN?
  • Which databases are shared between portal clusters and WAS servers?
  • Are the cluster nodes on the same subnet as WAS servers?
  • Is source code review in scope for the portlet code on the nodes? (the "code viewable" stuff)
